# CLAUDE.md

Operational guide for working in this repo. For the high-level project overview, stack, and
service responsibilities, see [CONTEXT.md](./CONTEXT.md) and [FEATURES.md](./FEATURES.md). This
file focuses on **how to build/run/deploy without breaking things** and the **non-obvious gotchas**.

Philo is a Photo & Timelapse bot that runs on a Raspberry Pi: it schedules a daily sunset
timelapse, captures frames with `rpicam-still`, stitches them with `ffmpeg`, and uploads/backs
them up to GitHub (per-frame) and a network/CIFS drive.

## Commands

```bash
npm start          # dev: tsx --env-file=.env src/main.ts   (runs from SOURCE)
npm run build      # tsc -b  -> compiles src/ into dist/     (production artifact)
npm run built      # node --env-file=.env dist/main.js       (runs the COMPILED build)
npm run production # build && built
npm run lint       # oxlint  (keep at 0 warnings)
npm test           # vitest
npm run testm -- still   # manual scratch tests (src/test.ts): upload | backup | lapse | still
npm run download-backup  # clones the sunset repos from GitHub locally (src/downloadBackups.ts)
npm run move-backup      # rsyncs local sunset dirs to FOLDER_BACKUP_MOUNT (src/moveBackups.ts)
```

## Deployment (read this before shipping)

The Pi runs the **compiled `dist/main.js` under PM2** (`ecosystem.config.cjs`, app name
`philopho`, `--env-file=.env`, `watch: true`). PM2 **watches `dist/` and ignores `src/`** â€” so
editing/uploading `src` does nothing until something compiles it to `dist`.

Code reaches the Pi via the **VS Code SFTP extension** (`.vscode/sftp.json`, `uploadOnSave: true`).
- Default target `pi5` = `pi@192.168.178.44:/home/pi/philo` (profiles also exist for pi3/pi4).
- **The Pi's `~/philo` is NOT a git checkout** (no `.git`) â€” it was copied via SFTP. `git pull`
  on the Pi will fail. Don't rely on git there.

### Gotchas that will bite you
- **`uploadOnSave` only fires on real editor saves.** Files written by tooling (CLI edits) or
  generated by `tsc` are *not* auto-uploaded. After a build, push explicitly:
  VS Code â†’ Command Palette â†’ **"SFTP: Sync Local â†’ Remote"** (or right-click `dist/` â†’ Upload Folder).
- **`npm run build` must succeed or `dist` is never emitted.** `tsc -b` fails the whole build on
  any type error, including in scratch files. `src/test.ts` is in the build graph â€” a broken
  scratch script there silently blocks every deploy and leaves `dist/` stale. Keep it compiling.
- **`dist/` can drift stale.** `tsc` does not delete orphaned outputs; old files from previous
  architectures linger. If something looks wrong, `npm run clean-build && npm run build`.

### Two recommended deploy flows
- **No-Pi-login (matches PM2 watch):** `npm run build` locally â†’ SFTP **Sync `dist/` â†’ Remote** â†’
  PM2 auto-restarts `philopho`. Verify with `pm2 logs philopho`.
- **Source + Pi build:** SFTP-upload `src/` â†’ on Pi `cd ~/philo && npm run build && pm2 restart philopho`.

> **Unresolved:** the Pi also has `philo-optic` and `philo-optic.service`, which are **not in this
> repo**. Before trusting any capture-side change, confirm what actually runs the sunset:
> `pm2 list` and `systemctl status philo-optic`. If `philo-optic` is the active capturer, changes
> to this repo's capture code won't take effect.

## Capture pipeline (sunset timelapse)

`Director.scheduleSunset` â†’ `Director.timelapse` â†’ `Timelapse.shoot` â†’ `Camera.photo` â†’
`StillCamera.takeImage` (`rpicam-still`). Frames are emitted per-capture and consumed by
`Producer` (Telegram preview + per-frame GitHub upload). At the end, `Timelapse` stitches with
`ffmpeg` (`lib/ffmpeg.ts`).

- Defaults (`.env`): **540 frames, 15000 ms interval**, scheduled ~1.2 h before sunset (golden hour).
- `Timelapse.shoot` drives capture with a fixed-rate `setInterval` (libcamera/rpicam crashes on
  long native timelapses, so frames are scheduled manually one at a time).
- **`frame` only advances on a successful capture; on error it is NOT incremented**, so the next
  tick retries the *same* frame number â€” this is what keeps the sequence gap-free. Error counting
  is **consecutive** (reset on success); only a stuck camera (>3 in a row) aborts the run.
- `Camera` holds a **mutex** (`#mutex`) and *throws* `Camera is busy` if a capture is requested
  while one is in flight â€” overlapping `setInterval` ticks hit this and are retried, not run concurrently.

### `spawnPromise` â€” shared, deliberately lenient (do not "fix" globally)
`lib/spawn.ts` `spawnPromise` is shared by `rpicam-still`, `rsync`, `ssh`, and `df`/`du`. It
**resolves on stdout `close` and does NOT check the process exit code**, and defaults to
`allowError = true` (ignores stderr). ffmpeg deliberately uses a *separate* wrapper
(`spawnPromisePrependStdErr` in `lib/ffmpeg.ts`) because it abuses stderr for progress.
- Past attempts to make `spawnPromise` strict broke other callers. **Scope capture-specific
  checks to the camera path, not the shared util.**
- Because failures are invisible to `spawnPromise`, the camera path verifies success itself:
  `StillCamera.takeImage` stats the output file and throws if it's missing/empty (see below).

## The dropped-frames bug (diagnosed + fixed, for context)

Symptom: sunset folders missing a handful of frames (e.g. 535â€“539 of 540), with **isolated holes
in the middle** while the run still reaches frame 540; `ffmpeg` renders truncate at the first gap.

- The holes are **capture-side**: `rpicam-still` occasionally fails (sensor hiccup, timeout,
  "failed to start camera"), exits non-zero, and writes **no file** â€” but `spawnPromise` resolved
  anyway (no exit-code check), so the loop counted it a success and advanced past a missing frame.
- Onset (from `N:\SunsetBackup` history): clean through Aug 2025, first holes **Sep 2025**,
  near-daily from **Nov 2025**. This tracks the **rpi5 hardware migration going flaky**, NOT the
  Dec-2025 `rpicam` switch (holes predate it) and NOT the Aug-2025 spawn edits (net behavior was
  unchanged). The silent-failure path was always latent; flaky hardware made it manifest.
- **Fix:** `StillCamera.takeImage` verifies a non-empty output file landed (throws otherwise);
  `Timelapse.shoot` retries the same frame on failure with consecutive-error counting + a
  per-failure log (`Frame [N] capture failed, retrying same frame â€¦`). Grep a night's `pm2 logs`
  for `capture failed` to quantify hardware hiccups.

## Storage / backup model (so you read the data right)

- **Capture+upload dir** = the per-day repo (`sunset-YYYY-MM-DD`), captured into and pushed to
  GitHub **per-frame** via `isomorphic-git` (`Git.upload`: add â†’ commit â†’ push; errors are
  swallowed and logged, no retry).
- **`downloadBackups.ts`** clones those repos back from GitHub (**shallow, `depth: 1`**) and logs
  `Storage has N files in <repo>` â€” so that count reflects **GitHub** content, not the Pi's disk.
- **`moveBackups.ts`** rsyncs local day-dirs to `FOLDER_BACKUP_MOUNT` with `--exclude=.git`,
  verifies file count + md5, and deletes local only after verifying (keeps the last 6 days). It
  compares local-vs-copy only, never against GitHub.
- `N:\SunsetBackup\*` on the dev machine are **git clones from GitHub** (they contain `.git`),
  i.e. they mirror GitHub, not raw local capture. Folders with `>540` files are days with **two
  runs** in one dir (false "extra" â€” not real duplicates of a single run).

## Conventions

- Keep `npm run lint` at **0 warnings** and `npm run build` green â€” a broken build silently
  blocks deploys.
- Prefer fixing the camera/capture path over the shared `spawnPromise`.
- `.env` is required and not in git; it lives on the Pi independently of uploaded code.
